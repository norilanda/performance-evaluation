def calculate_access_overhead(df):
    values = df["timestamp_ns"]
    adjacent_differences = values.diff().dropna()
    sum_of_adjacent_differences = adjacent_differences.sum()
    average_value = sum_of_adjacent_differences / len(adjacent_differences)

    print(f"Number of adjacent differences: {len(adjacent_differences)}")
    print(f"Sum of adjacent differences: {sum_of_adjacent_differences}")
    print(f"Average of access overhead: {average_value} ns")

The resolution of the timers (related to units returned)


To determine the resolution we can use the Frequency field of the timer: https://learn.microsoft.com/en-us/dotnet/api/system.diagnostics.stopwatch.frequency?view=net-10.0#system-diagnostics-stopwatch-frequency. The documentation says that:

"The Frequency value depends on the resolution of the underlying timing mechanism. If the installed hardware and operating system support a high-resolution performance counter, then the Frequency value reflects the frequency of that counter. Otherwise, the Frequency value is based on the system timer frequency."

It means that the result depend on the underlying system.
To deternime resolution, we will print to console value of the frequency field. In my case it is equal to 10_000_000 ticks per second. 1 second contains 1 billion nanoseconds, so it means that the resolution of a timer is 100ns. This is also confirmed experimentaly - all the measured time differences are dividable by 100.

The granularity of the timers (related to minimum increment)


The minimal increment is 100 ns, so the granularity is 100 ns.

The overhead of the timer access (related to minimum delta)


The overhead of the timer access is smaller than the resolution, because we can get the same timer value during a couple of accesses in a sequence. It is worth to mention that this accesses number for which the results do not change is not stable - sometimes it is 3, sometimes it is 4 or even 6.
Therefore, to calculate the overhean, we will sum up all the differences and divide them by their number. This will allow us to gain the average overhead. The value was presented in the above calculations.

The stability of the timer (both during and after warm up)


We can see that the timer is stable during and after warmup except the last measurments.

The overflow period of the timer


The inspected method returns long value that represents the numbers of ticks. The max value of long is 9,223,372,036,854,775,807 (https://learn.microsoft.com/en-us/dotnet/csharp/language-reference/builtin-types/integral-numeric-types)

The resolution of the timers (related to units returned)


The resolution for the timer depends on the value type SYSDATETIME function can return. It returns DATETIME2(7): https://learn.microsoft.com/en-us/sql/t-sql/data-types/datetime2-transact-sql?view=sql-server-ver17 which is stored in a temporary table variable. 7 in the DATETIME2(7) indicates the precision. In the documentation we see Accuracy 100 nanoseconds, so the resolution for the timer is 100 nanoseconds.
In the experimental results we see that all the numbers are dividable by 100 but not 1000, so it prooves that the resolution is 100 ns.

The granularity of the timers (related to minimum increment)



Let's find the minimal increment:

def calculate_minimum_increment(df):
    values = df["timestamp_ns"]
    adjacent_differences = values.diff().dropna()
    adjacent_differences = adjacent_differences[adjacent_differences > 0]
    min_of_adjacent_differences = adjacent_differences.min()

    print(f"Minimum of adjacent differences: {min_of_adjacent_differences}")

The overflow period of the timer


The DATEDIFF_BIG returns Signed bigint value which can store 9,223,372,036,854,775,807 as a maximum possible value. So to go from 0 to it we will need the similar number of years as for .NET method. https://learn.microsoft.com/en-us/sql/t-sql/functions/datediff-big-transact-sql?view=sql-server-ver17#return-type

As for the DATETIME2, the maximum posible date it can store is 9999-12-31 https://learn.microsoft.com/en-us/sql/t-sql/data-types/datetime2-transact-sql?view=sql-server-ver17

So these limits will not allow us to check what will happen during overflow experimentally.
